Dokumentasi Sistem Monitoring Guru 1. Arsitektur & Teknologi Utama • Backend / Web App: Laravel 11 (PHP 8+) • Mobile App: Flutter (Dart) • Database: MySQL  • Autentikasi API: Laravel Sanctum (Token-based) 2. Struktur Pengguna (Role-based System) Sistem menggunakan sistem satu pintu pada tabel users. Pemisahan hak akses diatur melalui kolom jabatan. Terdapat 3 peran (Role) utama: 1. Admin (jabatan = admin): Bisa mengakses Web Admin Panel untuk CRUD Master Data (Mapel, Kelas, Ruangan, Guru, Jadwal Ajar). 2. Guru Pengajar (jabatan = guru): Bisa mengakses Web Portal Guru dan Aplikasi Mobile. Bertugas untuk melakukan scan QR Absen Masuk & Keluar di kelas, serta mengecek riwayat mengajarnya. 3. Ketua Kelas (jabatan = ketuakelas): Hanya bisa mengakses Aplikasi Mobile (dengan f itur terbatas). Bertugas memastikan/memverifikasi absen yang dilakukan di kelasnya. 3. Alur Kerja (Workflow) Absensi QR Code Inti dari sistem ini adalah Absensi Berbasis Lokasi & Jadwal (QR Code). A. Persiapan Jadwal (Oleh Admin) 1. Admin menambahkan Master Data: Kelas, Ruangan, Mapel, dan Akun Guru. 2. Admin membuat Jadwal Ajar. Jadwal ini bersifat Weekly Recurring (berulang setiap minggu). • Contoh: "Matematika", Guru "Pak Budi", Hari "Senin", "08:00 - 10:00" di "Ruang 1". B. Proses Absen (Oleh Guru via Mobile/Web) 1. Guru login ke Aplikasi Mobile (atau Web Portal). 2. Sistem mendeteksi Hari Ini (misal: Senin) dan menarik semua jadwal Guru tersebut di hari Senin. 3. Peran Ketua Kelas (Penyedia QR Code): • Ketua Kelas login ke aplikasi (Web atau Mobile). • Di dashboard mereka, Ketua Kelas menekan tombol "Generate QR". • Sistem akan memunculkan sebuah QR Code dinamis di layar perangkat Ketua Kelas yang menyimpan informasi validasi kelas dan ruangan saat itu. 4. Alur Scan (Oleh Guru): • Jika guru belum absen hari ini di jadwal tersebut: Tombol menjadi Scan QR MASUK. Guru memindai QR Code yang ada di layar perangkat Ketua Kelas. Sistem memverifikasi kecocokan Ruangan dan membuat rekaman di tabel absen_masuks. • Jika guru sudah Scan Masuk: Tombol berubah menjadi Scan QR KELUAR. Di akhir jam pelajaran, guru memindai QR Code yang sama. Sistem mencatat jam keluar di tabel absen_keluars. • Setelah selesai: Jadwal akan berstatus Selesai. C. Transparansi Riwayat (Per Mapel) • Di aplikasi mobile maupun web, Guru dapat mengklik Mata Pelajaran di dashboard untuk masuk ke layar "Riwayat Mengajar". • Layar ini akan melakukan query (API GET /riwayat-mapel/{mapel_id}) untuk menarik semua rekaman AbsenMasuk dan AbsenKeluar masa lalu untuk mata pelajaran tersebut. 4. Struktur Database Inti • users: Tabel sentral untuk Admin, Guru, dan Ketua Kelas. Login menggunakan NIK yang bersifat unique. • Kolom penting: nik, password, jabatan (admin/guru/ketuakelas), kelas_id (khusus ketua kelas). • jadwal_ajars: Template jadwal mingguan. Relasi ke users (sebagai guru_id), mapels, kelas, dan ruangans. • Kolom: hari, jam_mulai, jam_selesai. • absen_masuks: Tabel log absensi harian (Masuk). • Kolom: tanggal, jam_masuk, jadwal_ajar_id. • absen_keluars: Tabel log absensi harian (Keluar). • Relasi: absen_masuk_id (1-to-1 dengan tabel Masuk), jam_keluar. 5. Endpoints API Mobile (Flutter) Semua rute berawalan /api/. Autentikasi membutuhkan Header Authorization: Bearer {token}. • POST /login: Validasi NIK dan password, mengembalikan Token Sanctum. • POST /logout: Menghapus token di database. • GET /jadwal: Menarik daftar jadwal hari ini untuk user yang login, lengkap dengan status apakah ia sudah absen hari ini. • POST /scan: Memproses raw data hasil pembacaan Kamera QR Code (Validasi ruangan dan membuat/mengupdate record absen). • GET /riwayat-mapel/{mapel_id}: Menarik sejarah mengajar (history absen) untuk suatu mapel.

Simulasi Data Tanpa Mutasi: Jika UI memerlukan data untuk dirender atau diuji, JANGAN PERNAH membuat script yang mengeksekusi query SQL (seperti INSERT, UPDATE, atau DELETE). Gunakan murni Mock Data (JSON statis) di level lokal Flutter untuk merepresentasikan response API.
